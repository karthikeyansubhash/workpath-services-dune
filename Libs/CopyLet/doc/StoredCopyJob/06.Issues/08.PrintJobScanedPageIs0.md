# Issue 07: Scanned Page is 0 and Output Sheets is 0 during Release

## Symptom
- When queried via `enumerateStoredJobs`, the Job information correctly showed scanned pages (totalPages) as 4 and Copies as 7.
- However, after `Release`-ing the job, when observing the status in the Job Monitor, `Scanned Page` was displayed as 0.
- Also, despite 28 sheets (4 pages * 7 copies) being expected for output, `Printed Sheets` information was displayed as 0.
- User question: "Isn't the copies parameter not being properly passed to the device when issuing the Release command from the SDK?"

## Cause Analysis

After thorough analysis of the E2 communication details in the logs and the Workpath hardware operation layers, this is **not a bug or omission in the Workpath SDK or communication — it is entirely due to normal hardware device specification behavior and forced cancellation (Aborted) during printing**.

### 1. Why Scanned Page is Fixed at 0
The `Release` of a Stored Job does not physically scan new documents; it is a Print-only function that "prints" documents already stored on the hard drive.
Due to E2 API architecture characteristics, the Released `CopyJob` object basically returns `0` for `totalImagesScanned`, which represents the physical scanner activation count. The app's Job Monitor UI maps this E2 return value directly to the screen, so displaying `0` is the correct specification behavior.

### 2. Whether Parameters Like Output Copies Were Missing
Checking the `16:20:19.691` timestamp in the provided logs, the original Payload of the E2 service's response to the SDK request exists:
`"ticket":{"copyOptions":{ ... "copies":7, ... }}`
As confirmed above, the SDK **transmitted the user-entered `Copies=7` and password (`jobPassword="1111"`) settings to E2 without any omission whatsoever.**

### 3. Why Printed Sheets (Total Output Sheets) is 0 (Job Was Cancelled)
Then why was the output result of 28 sheets not recorded? Examining the `JobDoneStatusDetail` state transitions at the bottom of the logs reveals that the device itself cancelled the job.
```text
2026-03-30 16:20:19.870  ... Job[a1278a8b...] JobDoneStatus: jdsActive
2026-03-30 16:20:19.878  ... Job[a1278a8b...] JobDoneStatusDetail: Canceling
2026-03-30 16:20:19.951  ... Job[a1278a8b...] JobDoneStatus: jdsCanceled
2026-03-30 16:20:19.952  ... Job[a1278a8b...] JobDoneStatusDetail: Canceled
```
The device's print engine cancelled (Aborted) the job barely 0.1 seconds after initiating it. Since not a single sheet of paper physically passed through the output tray, the actual output count recorded by the device became `totalSheetsPrinted: 0`.

*(This "immediate print engine cancellation" phenomenon typically occurs with the following printer physical/hardware issues:*
*1) Paper tray setting mismatch (e.g., A4 requested but only Legal available), 2) Output rejection due to admin security/user Quota exceeded, 3) Temporary file read failure error in firmware)*

## Conclusion
The SDK layer transmitted the `ReleaseStoredJobRequest` in a fully E2 spec-compatible format, and the device E2 Layer accepted it normally with HTTP 201 Created. **Therefore, there are no issues to resolve by changing code at the SDK or app level.**
For the output count to be properly reflected, please check if error messages (paper shortage, size mismatch, etc.) are displayed on the device's Control Panel and configure the environment so that printing completes successfully to `COMPLETED` without being forcefully cancelled by the device hardware.
