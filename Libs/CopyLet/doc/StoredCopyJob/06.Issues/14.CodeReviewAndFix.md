## 1. StoredCopyJobAdapter.buildScanOptions — Missing convertWtoE Default Values

### Problem
In `StoredCopyJobAdapter.buildScanOptions()`, `null` was passed as the second argument (default value) when calling `CopyTypeMapping.xxx.convertWtoE()`.
In `CopyJobAdapter.getCopyJobTicket()`, `DefaultOptions` values were passed to provide fallback on mapping failure, but this pattern was missing in `StoredCopyJobAdapter`.

### Changes
- Added `DefaultOptions defaultOptions` parameter to `createScanJobForStorage()` signature
- Passed `DefaultOptions` to `buildScanTicket()` and `buildScanOptions()`
- Changed 9 `convertWtoE()` calls from `null` → `defaultOptions.getXXX()` (null-safe)
- For `originalPlexMode`, passed `new Pair<>(defaultOptions.getOriginalPlexMode(), defaultOptions.getMediaBindingFormat())`

### Modified Files
| File | Changes |
|------|---------|
| `StoredCopyJobAdapter.java` | Added `DefaultOptions` parameter, passed default values |
| `CreatingCopyJobState.java` | Added `copyJobService.getDefaultOptions(packageName)` call and passed result |
| `StoredCopyJobAdapterTest.java` | Updated test code for signature change |
| `StoredCopyJobAdapterInstrumentedTest.java` | Updated test code for signature change |

## 3. CopyJobCompletedState.deleteStoredJobOnRelease — Preventing Unnecessary Calls

### Problem
In `CopyJobCompletedState.processCompletedJob()`, `deleteStoredJobOnRelease()` was called for all job types (Copy, Store, Release).
Although there was a `KEY_DELETE_ON_RELEASE` guard inside the method ensuring actual deletion only occurs for Release, unnecessary method calls and Bundle reads occurred during Copy/Store jobs.

### Changes
- In `processCompletedJob()`, check `KEY_DELETE_ON_RELEASE` flag first, then conditionally call `deleteStoredJobOnRelease()`
- Added `Bundle extraBundle` parameter to `deleteStoredJobOnRelease()` signature (removed duplicate Bundle lookup)

### Modified Files
| File | Changes |
|------|---------|
| `CopyJobCompletedState.java` | Call deleteStoredJobOnRelease only for Release jobs |

## 4. processStoredCopyJobIntent — Removing Unnecessary save → replaceKey Step

### Problem
In `CreatingCopyJobState.processStoredCopyJobIntent()`:
1. `saveCopyAttributes(context, rid, copyAttributes)` — Save with rid as key
2. `createScanJobForStorage()` — On failure, clean up with `remove(context, rid)`
3. `replaceKey(context, rid, scanJobId)` — On success, change key to scanJobId

In the pattern of saving with rid key then replacing with scanJobId, the rid key was never used between save and replaceKey.

### Changes
- Perform ScanJob creation first, and only on success call `saveCopyAttributes()` directly with `scanJobId` as key
- Removed `replaceKey()` call
- On failure, preference cleanup (`remove`) unnecessary (not yet saved)

### Before
```java
// 1. Save with RID
StoredCopyJobPreferenceStorage.saveCopyAttributes(context, rid, copyAttributes);
// 2. Create ScanJob
String scanJobId = StoredCopyJobAdapter.createScanJobForStorage(...);
if (scanJobId == null) {
    StoredCopyJobPreferenceStorage.remove(context, rid);  // Cleanup on failure
    return error;
}
// 3. Replace key
StoredCopyJobPreferenceStorage.replaceKey(context, rid, scanJobId);
```

### After
```java
// 1. Create ScanJob
String scanJobId = StoredCopyJobAdapter.createScanJobForStorage(...);
if (scanJobId == null) {
    return error;  // Not yet saved, no cleanup needed
}
// 2. Save directly with scanJobId
StoredCopyJobPreferenceStorage.saveCopyAttributes(context, scanJobId, copyAttributes);
```

### Modified Files
| File | Changes |
|------|---------|
| `CreatingCopyJobState.java` | Simplified save → createScanJob → replaceKey flow to createScanJob → save |

## 6. @JsonInclude(NON_NULL) Annotation Review

### Review Targets
- `ReleaseStoredJobRequest.java`: `@JsonInclude(JsonInclude.Include.NON_NULL)` added
- `RemoveStoredJobRequest.java`: `@JsonInclude(JsonInclude.Include.NON_NULL)` added
- `ReleaseStoredJobRequest_SolutionContext_Value.java`: No change (same as master)

### Review Result: Keep (Required)
- `ReleaseStoredJobRequest`: `jobPassword` and `solutionContext` are optional fields. When null, including `"jobPassword": null` in JSON could be misinterpreted by E2 API as setting an empty password
- `RemoveStoredJobRequest`: `jobPassword` is null for stored jobs without password. Including null in JSON is unnecessary
- Other E2 Request types don't have this annotation, but those types either have no optional fields or all fields are always set
- Although auto-generated files, this modification is necessary for correct interaction with the E2 API

### Changes: None (keep current state)

## 7. OXPCopyletContentProvider — Remove Direct E2 Type References

### Problem
`OXPCopyletContentProvider` was directly depending on E2 types by using `import com.hp.ext.service.copy.StoredJobs`.
The ContentProvider layer is a class that should not have dependencies on E2 types, so it needed to be modified to not directly reference E2 types.

### Changes
- Added new facade method `enumerateAndConvertStoredJobs()` to `CopyJobAdapter`
  - Internally calls `copyJobService.enumerateStoredJobs()` and performs `convertToStoredJobInfoList()` conversion
  - Callers receive only `List<StoredJobInfo>` without needing to know the E2 type (`StoredJobs`)
- Removed `import com.hp.ext.service.copy.StoredJobs` from `OXPCopyletContentProvider`
- Changed `ENUMERATE_JOBS` case to call `CopyJobAdapter.enumerateAndConvertStoredJobs()`

### Before
```java
// OXPCopyletContentProvider.java
import com.hp.ext.service.copy.StoredJobs;  // ← E2 dependency

case Copylet.Method.ENUMERATE_JOBS: {
    StoredJobs storedJobs = copyDeviceService.enumerateStoredJobs(appPackageName);
    List<StoredJobInfo> storedJobInfoList =
            CopyJobAdapter.convertToStoredJobInfoList(getContext(), storedJobs, copyDeviceService, appPackageName);
    ...
}
```

### After
```java
// OXPCopyletContentProvider.java — No E2 imports

case Copylet.Method.ENUMERATE_JOBS: {
    List<StoredJobInfo> storedJobInfoList =
            CopyJobAdapter.enumerateAndConvertStoredJobs(getContext(), copyDeviceService, appPackageName);
    ...
}
```

### Modified Files
| File | Changes |
|------|---------|
| `CopyJobAdapter.java` | Added `enumerateAndConvertStoredJobs()` facade method |
| `OXPCopyletContentProvider.java` | Removed E2 import, using facade method |

## 8. Delete Job — Changed from StateMachine to Direct ContentProvider Call

### Review Comment
> "Deleting a stored job is a stateless operation, so it should not go through the StateMachine.
> Please follow the design guideline: ContentProvider → CopyJobAdapter → DeviceCopyJobService."

### Analysis Result: Reviewer is correct

**Why Delete is Stateless:**
1. `processDeleteJobIntent()` transitions to `EndState` immediately after E2 deleteStoredJob API call — no monitoring/state transition needed
2. StateMachine setup like `setJobBundle`, `setJobRid`, `setTargetPackageName` is unnecessary
3. IntentService async processing + ResultReceiver + CountDownLatch synchronization mechanism is unnecessarily complex

**No Impact on CopyJobCompletedState.deleteStoredJobOnRelease:**
- The auto-delete flow after Release occurs within the StateMachine (part of Release monitoring)
- Does not use `processDeleteJobIntent`, calls `copyJobService.deleteStoredJob()` directly
- Not affected by this change

### Before (ContentProvider → IntentService → StateMachine → E2)
```java
// OXPCopyletContentProvider.deleteJob()
CountDownLatch countDownLatch = new CountDownLatch(1);
ResultReceiver resultReceiver = new ResultReceiver(...) { ... };
CopyJobIntentService.startDelete(getContext(), reqBundle, resultReceiver);
countDownLatch.await(60, TimeUnit.SECONDS);  // ← always times out (BUG)

// CreatingCopyJobState.processDeleteJobIntent()
RemoveStoredJobRequest removeRequest = new RemoveStoredJobRequest();
copyJobService.deleteStoredJob(packageName, storedJobId, removeRequest);
StoredCopyJobPreferenceStorage.remove(context, storedJobId);
return new EndState();  // ← terminates without returning result
```

### After (ContentProvider → CopyJobAdapter → E2)
```java
// OXPCopyletContentProvider.deleteJob()
DeleteRequestIntent.IntentParams reqParams = DeleteRequestIntent.getIntentParams(reqBundle);
JobCredentialsAttributes credentials = reqParams != null ? reqParams.getJobCredentialsAttributes() : null;
String packageName = reqParams != null ? reqParams.getPackageName() : null;
CopyJobAdapter.deleteStoredJob(getContext(), copyDeviceService, packageName, jobId, credentials);

// CopyJobAdapter.deleteStoredJob() — NEW
RemoveStoredJobRequest removeRequest = new RemoveStoredJobRequest();
applyDeleteJobPassword(removeRequest, credentials);
StoredJob_Remove removeResult = copyJobService.deleteStoredJob(packageName, storedJobId, removeRequest);
StoredCopyJobPreferenceStorage.remove(context, storedJobId);
```
