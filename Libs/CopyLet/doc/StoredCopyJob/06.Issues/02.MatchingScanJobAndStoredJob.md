# StoredCopyJob Release Flow 분석 결과

## 1. Release 시 `storedJobAttributes`와 `savedCopyAttributes` 획득 과정

### storedJobAttributes (✅ 정상)
- **출처**: 앱이 `CopierService.releaseStoredJob(context, storedJobAttributes)`를 호출할 때 전달
- **전달 경로**: `ReleaseRequestIntent` → `BroadcastReceiver` → [CopyJobIntentService](file:///D:/_Work/Android/sdk-dune/Libs/CopyLet/src/main/java/com/hp/jetadvantage/link/services/copylet/service/CopyJobIntentService.java#27-132) → `CreatingCopyJobState.processReleaseJobIntent()`
- **포함 정보**: `storedJobId`, `copies`, `jobCredentials` (password)
- **로그 확인**: `storedJobId=59ecaac2-...`, `copies=1` → 정상적으로 수신됨

### savedCopyAttributes (❌ 항상 null)
- **출처**: `StoredCopyJobPreferenceStorage.getCopyAttributes(context, storedJobId)`
- **설계 의도**: Create 시 저장된 원본 CopyAttributes를 Release 시 복원하여 사용자가 설정했던 복사 옵션을 그대로 적용
- **현재 동작**: **항상 null 반환** → DefaultOptions fallback 사용

## 2. 🐛 버그 분석: Key 불일치 문제

### Create 시 키 저장 흐름 ([CreatingCopyJobState.java:236-254](file:///D:/_Work/Android/sdk-dune/Libs/CopyLet/src/main/java/com/hp/jetadvantage/link/services/copylet/service/CreatingCopyJobState.java#L236-L254))

```
1. saveCopyAttributes(context, RID, copyAttributes)     // key = RID (UUID)
2. E2 ScanJob 생성 → scanJobId 반환
3. replaceKey(context, RID, scanJobId)                   // key 교체: RID → scanJobId
```

- **최종 저장 키**: `scanJobId` (E2 ScanJob ID)

### Release 시 키 조회 흐름 ([CreatingCopyJobState.java:338](file:///D:/_Work/Android/sdk-dune/Libs/CopyLet/src/main/java/com/hp/jetadvantage/link/services/copylet/service/CreatingCopyJobState.java#L338))

```
getCopyAttributes(context, storedJobId)                  // key = storedJobId (E2 StoredJob ID)
```

- **조회 키**: `storedJobId` (앱이 [enumerateStoredJob](file:///D:/_Work/Android/sdk-dune/Libs/JetAdvantageLinkApi/src/main/java/com/hp/jetadvantage/link/api/copier/CopierService.java#406-453)에서 받은 E2 StoredJob ID)

### 🔑 핵심 문제점

| 단계 | 키 값 | 출처 |
|------|--------|------|
| Create 저장 | `scanJobId` (E2 ScanJob에서 반환) | `StoredCopyJobAdapter.createScanJobForStorage()` |
| Release 조회 | `storedJobId` (E2 enumerateStoredJob에서 반환) | `CopierService.enumerateStoredJob()` |

**`scanJobId`와 `storedJobId`가 동일한 값임을 보장하는 메커니즘이 없습니다.**

E2 플랫폼에서 "Scan to JobStorage" 작업이 완료되면, E2가 내부적으로 해당 스캔 결과를 `storedJob`으로 등록합니다. 이때 E2가 반환하는 `storedJobId`는 원래의 `scanJobId`와 **동일할 수도 있고, 다를 수도 있습니다.** 로그를 보면:

- Create 시 `scanJobId` (PreferenceStorage에 저장됨) — 로그에 표시되지 않음
- Release 시 `storedJobId = 59ecaac2-5ac0-4873-9d04-b5b29252727c` (enumerateStoredJob에서 획득)
- PreferenceStorage에서 `key=59ecaac2-...`를 찾지 못함 → **키 불일치 확인**

## 4. 최종 분석 및 결론 (✅ 문제 해결됨)

가장 최근에 추출된 전체 로그(`02-1.StoredJocAllLogs.txt`)를 통해 Create부터 Release까지의 모든 흐름을 확인한 결과, **E2 플랫폼은 ScanJob 생성 시 발급된 `jobId`를 해당 작업이 완료된 후 생성되는 StoredJob의 `jobId`로 동일하게 사용함**을 명확히 확인했습니다.

### 로그 증거
1. **Create 완료 시 (ScanJob ID 반환):**
   ```
   12:13:53.588 createScanJobForStorage : ScanJob created, jobId=2590f052-6171-463c-b0a6-1873982b663a
   12:13:53.589 replaceKey : oldKey=17b37770-... → newKey=2590f052-6171-463c-b0a6-1873982b663a
   ```
2. **Enumerate Jobs 시 (StoredJob 포함 확인):**
   ```
   12:14:07.758 StoredJobList=[...,storedJobId=2590f052-6171-463c-b0a6-1873982b663a,storedJobName=2026033003135171,...]
   ```
3. **Release 시 (PreferenceStorage 정상 조회):**
   ```
   12:14:24.041 getCopyAttributes : key=2590f052-6171-463c-b0a6-1873982b663a
   12:14:24.128 processReleaseJobIntent : savedCopyAttributes=found
   ```

### 💡 결론
- **매핑 로직 불필요:** `scanJobId`와 `storedJobId`가 동일하게 유지되므로, 별도의 ID 매핑 로직을 구현할 필요가 없습니다.
- **원본 설계 정합성 확인:** Create 시 `scanJobId`를 키로 `CopyAttributes`를 내부 디바이스 저장소에 저장하고, Release 시 `storedJobId`로 꺼내어 인쇄 시 적용하는 기존 아키텍처는 E2 동작 스펙과 100% 일치합니다.
- **이전 실패 원인:** 이전의 "Key 불일치" 의심은, Create 과정에서 `folderName` 및 `jobPasswordType` 누락으로 인한 **HTTP 400 Bad Request** 때문에 실제 ScanJob 생성이 정상 완료되지 못해 생긴 혼선이었습니다. (현재 해당 파라미터 버그 모두 수정됨)

이로써 **Store & Release 흐름 상의 모든 이슈가 해결되었습니다.**
