# Issue #13: RetentionModeOnRelease DELETE Implementation

## Background

`CopyAttributes.RetentionMode` has 3 values: `DEFAULT`, `KEEP`, `DELETE`.
When `RetentionModeOnRelease` is set to `DELETE`, the Stored Job should be automatically deleted after a successful Release.

E2 API has a `storeJobDeleteOnRelease` property (see 12.E2_Workpath_Mapping_Analysis.md),
but since the current E2 mapping is commented out, the delete logic is implemented **at the CopyLet service level**.

## Analysis

### Data Flow
1. App → `CopyAttributes.Builder.setRetentionModeOnRelease(RetentionMode.DELETE)` → CopyAttributes created
2. Create STORE → `StoredCopyJobPreferenceStorage.saveCopyAttributes()` → CopyAttributes saved
3. Release → `StoredCopyJobPreferenceStorage.getCopyAttributes()` → savedCopyAttributes restored
4. Release success → Check RetentionMode → If DELETE, **store delete info in extraJobBundle**
5. Monitoring → Copy completed → **Actual deletion performed in CopyJobCompletedState**

### Existing Delete Functionality
`processDeleteJobIntent()` already uses the `copyJobService.deleteStoredJob()` + `StoredCopyJobPreferenceStorage.remove()` pattern.

### Approach
- `CopyAttributes.mStoredJobRetentionModeOnRelease` field is package-private, so accessed via `CopyAttributesReader`
- Delete failure should not affect Copy execution, so exceptions are caught and only logged
- For password-protected Stored Jobs, E2 requires the password even for deletion, so `RemoveStoredJobRequest` is configured with the password

## Issue History

### v1: Delete Immediately After Release (Problem Found)
- **Problem 1 (403 error):** Password was not set in `RemoveStoredJobRequest`, causing E2 to respond with 403
- **Problem 2 (timing):** Deleting immediately after Release (before Copy starts) → Device Critical Error occurred

### v2: Delete After Copy Completion (Current Implementation)
Both problems resolved:
1. **Password:** Extract password from `StoredJobAttributes.getJobCredentialsAttributes()` → Store in extraJobBundle → Set `RemoveStoredJobRequest.setJobPassword()` at deletion time
2. **Timing:** Deletion performed in `CopyJobCompletedState.processCompletedJob()` → Executed after MonitoringCopyJobState's Copy completion notification (MSG_REPORT_COMPLETED)

## Changes

### CreatingCopyJobState.java

**Change location:** `processReleaseJobIntent()` method

Instead of deleting immediately after Release, store delete info in extraJobBundle to pass to CopyJobCompletedState.

**Before:**
```java
// 5. Monitor the released copy job (E2 CopyNotification)
return new MonitoringCopyJobState(copyJobId);
```

**After:**
```java
// 5. Store delete info if RetentionModeOnRelease is DELETE (actual delete in CopyJobCompletedState)
storeDeleteOnReleaseInfo(stateMachine, savedCopyAttributes, storedJobAttributes, packageName, storedJobId);

// 6. Monitor the released copy job (E2 CopyNotification)
return new MonitoringCopyJobState(copyJobId);
```

**New method: `storeDeleteOnReleaseInfo()`**
```java
private void storeDeleteOnReleaseInfo(BaseJobIntentServiceStateMachine stateMachine,
        CopyAttributes savedCopyAttributes,
        StoredJobAttributes storedJobAttributes,
        String packageName, String storedJobId) {
    // Return if RetentionModeOnRelease != DELETE
    // Store 4 keys in extraJobBundle:
    //   KEY_DELETE_ON_RELEASE (boolean) = true
    //   KEY_DELETE_STORED_JOB_ID (String) = storedJobId
    //   KEY_DELETE_PACKAGE_NAME (String) = packageName
    //   KEY_DELETE_PASSWORD (String) = credentials.getStoreJobPassword()
}
```

### CopyJobCompletedState.java

**Changes:** After Copy completion, check delete info from extraJobBundle and perform actual deletion.

**Added constants:**
```java
static final String KEY_DELETE_ON_RELEASE = "deleteOnRelease";
static final String KEY_DELETE_STORED_JOB_ID = "deleteStoredJobId";
static final String KEY_DELETE_PACKAGE_NAME = "deletePackageName";
static final String KEY_DELETE_PASSWORD = "deletePassword";
```

**New method: `deleteStoredJobOnRelease()`**
```java
private void deleteStoredJobOnRelease(BaseJobIntentServiceStateMachine stateMachine) {
    // Check KEY_DELETE_ON_RELEASE from extraJobBundle
    // If true:
    //   1. Create StandardDeviceCopyJobService
    //   2. Set password in RemoveStoredJobRequest
    //   3. Call copyJobService.deleteStoredJob()
    //   4. Call StoredCopyJobPreferenceStorage.remove()
    // On exception, only log
}
```

### State Transition Flow

```
CreatingCopyJobState (Release)
  ├── Release API call success
  ├── storeDeleteOnReleaseInfo() → Store delete info in extraJobBundle
  └── → MonitoringCopyJobState (Copy progress monitoring)
           └── MSG_REPORT_COMPLETED (Copy completed)
                └── → CopyJobCompletedState
                       ├── processCompletedJob() : Set JobExecutionMode.STORE
                       └── deleteStoredJobOnRelease() : E2 delete + PreferenceStorage cleanup
                            └── → EndState
```

### Design Decision Rationale
1. **Delete after Copy completion** — Deleting immediately after Release causes Device Critical Error due to Stored Job reference errors during Copy. It is safe to delete only after Copy completion (MSG_REPORT_COMPLETED)
2. **Pass info via extraJobBundle** — Leverages the State Machine's existing mechanism. Passes storedJobId, packageName, password from CreatingCopyJobState to CopyJobCompletedState
3. **Include password** — E2 requires password even for individual delete API after Release authentication (403 Incorrect password). Extracted from `StoredJobAttributes.getJobCredentialsAttributes()` and passed along
4. **Catch exception + log only on delete failure** — Deletion is a side effect and should not affect Copy result reporting
5. **Call PreferenceStorage.remove() on successful E2 deletion** — Cleans up CopyAttributes stored with storedJobId key. On delete failure, PreferenceStorage is also preserved (can be cleaned up by app's explicit DELETE call later)
