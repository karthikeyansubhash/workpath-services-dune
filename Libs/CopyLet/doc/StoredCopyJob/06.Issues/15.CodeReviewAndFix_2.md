The JobId must be validated first before extracting ScanJobStatus and updating the state machine. Multiple scan jobs may be running concurrently, and only the JobId monitored by this state machine should be processed.

### Developer Comment
The reviewer's comment is accurate. The existing `MonitoringScanJobState` (ScanLet) already has JobId validation implemented, but this validation was missing in `MonitoringStoredCopyJobState` and `MonitoringCopyJobState`.

When multiple Scan/Copy Jobs run concurrently, the notification callback receives notifications for **all** Jobs. Without JobId validation, state changes from other Jobs could incorrectly update the state of the currently monitored Job. In severe cases, completion/failure notifications from other Jobs could cause premature termination or incorrect state transitions of the current Job.

Following the pattern from `MonitoringScanJobState`, added JobId validation to both files:
1. Extract the notification's JobId via `ScanJobNotificationContent.getScanJobId()` / `CopyJobNotificationContent.getCopyJobId()`
2. Compare with `UUID.fromString(jobId).equals(notifiedJobId.getValue())`
3. Ignore on mismatch (notification from another Job)
4. Transition to `MSG_REPORT_FAIL` on jobId parse failure (invalid UUID)
