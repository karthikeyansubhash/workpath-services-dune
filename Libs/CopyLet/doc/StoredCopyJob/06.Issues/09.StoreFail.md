# Issue 08: Stored Copy Job Creation Failure and Release Malfunction During Refactoring

## 1. Overview
During refactoring to eliminate duplicate code in `CreatingCopyJobState.java`, the `createReleaseCopyJobBundle()` method was deleted and merged into the existing `createCopyJobBundle()` method. However, the essential parameter `JobExecutionMode` was not properly set, causing scan/print operations to fail or behave unexpectedly.

## 2. Symptom
1. **Release Operation Failure (Unable to Start Scan):**
   Occurred when `createReleaseCopyJobBundle` was completely deleted and `createCopyJobBundle` was called directly.
   Even when release was requested from the app, no operation (scan/print) was initiated.
2. **Store Selection Proceeds as Normal Copy (Log 08-1.StoreFail.txt):**
   A Store job was instructed, but the generated Job Bundle went through the merged method and `JobExecutionMode.STORE` was omitted. As a result, the downstream State Machine misidentified the job as a normal print job and failed to follow the proper Store flow.

## 3. Cause Analysis
The two methods had approximately 85% identical code and appeared to be duplicates, but there were **critical semantic differences**:

1. **`JobExecutionMode` Setting Method:**
   - **`createCopyJobBundle`**: Uses the value from the parameter reader (`reader.getJobExecutionMode()`) as-is. If null is passed, the mode itself is not set and the default (`DEFAULT` or `NORMAL`) is applied.
   - **`createReleaseCopyJobBundle`**: Always **hardcodes** `JobExecutionMode.STORE` when called. (Because release always processes a stored job)

2. **Job Name Fallback:**
   - **`createCopyJobBundle`**: Uses `""` or `"[Untitled]"` when the name is null.
   - **`createReleaseCopyJobBundle`**: Uses `"[Stored Job]"` when the name is null.

During refactoring, simply merging into a single method caused the release call site (`processReleaseJobIntent`) to lose the forced Mode setting and fall into `DEFAULT` mode, resulting in State Machine malfunction.

## 4. Solution
Instead of a simple merge, improved the logic by adding a parameter that explicitly distinguishes the method call purpose (normal creation vs. Release).

1. **Parameter Addition:**
   Added the existing `CopyAttributes.JobExecutionMode` as an identification parameter to the API.
   `private Bundle createCopyJobBundle(int clientVersion, CopyAttributesReader copyAttributesReader, CopyAttributes.JobExecutionMode bundleMode)`

2. **Branch Handling:**
   Inside the method, behavior is distinguished based on the received `bundleMode`.
   - `bundleMode == STORE`: Release flow. Forces mode to `STORE` and uses `[Stored Job]` if no name is provided.
   - `bundleMode == NORMAL`: Create flow. Follows the reader's own mode and uses `[Untitled]` etc. if no name is provided.

3. **Call Sites Update:**
   Call sites can now express their purpose for bundle creation with high readability.
   - Normal Copy (Scan & Print): `createCopyJobBundle(..., JobExecutionMode.NORMAL)`
   - Stored Copy creation: `createCopyJobBundle(..., JobExecutionMode.NORMAL)`
   - Stored Copy release: `createCopyJobBundle(..., JobExecutionMode.STORE)` -> Release-specific hardcoded attributes applied

## 5. Conclusion
This fix eliminated code duplication (37 lines reduced) while perfectly preserving the semantics of bundle creation for each purpose (Create vs. Release), and significantly improved code readability. Both build and tests passed successfully.
