# Issue 04: Delete Stored Job Fails with "Incorrect password"

## Symptom
When attempting to delete a Stored Copy Job with a password set using the SDK, even though the correct password was entered, the following HTTP 403 authentication failure fault log occurs from the E2 device, causing the delete operation to fail.

```
2026-03-30 14:58:24.973 22260-23155 [WS]DSS/DeviceService    E  perform 2: OXPdHttpRequestException=HTTP request failed. 403: {"errors":[{"code":"noAuthorityError","errorDetails":{"typeGUN":"com.hp.ext.types.baseTypes.version.1.type.noAuthorityError","value":{}},"message":"Incorrect password for stored job"}],"httpStatusCode":403,"version":"1.0.0"} (Ask Gemini)
```

## Cause
Within the Workpath SDK internals, the `jobPassword` was not being passed to the E2 REST client during the Delete processing flow, and an empty Body (`{}`) was being sent to the E2 `/ext/copy/v1/copyAgents/{agentId}/storedJobs/{jobId}/remove` API.

1. `RemoveStoredJobOperationResourceFacade` was hardcoded to always send an empty `StoredJob_Remove` object (without a password field) as the HTTP Request Body.
2. The `deleteStoredJob()` Service interface called in the `processDeleteJobIntent()` step within the `CreatingCopyJobState` class did not have a signature that accepts `RemoveStoredJobRequest`.
3. As a result, even though the user entered the password in the app UI and it was passed all the way to `DeleteRequestIntent`, it was omitted during the final E2 call.

## Solution
Modified **4 layers** to enable passing the `RemoveStoredJobRequest` object with the password field in the Request Body when calling the E2 Remove API.

1. **Model Layer (`RemoveStoredJobRequest.java`)**
   - Added `@JsonInclude(JsonInclude.Include.NON_NULL)` annotation at the top of the class to allow empty JSON to be parsed when the password is omitted.
2. **REST Client Layer (`RemoveStoredJobOperationResourceFacade.java`)**
   - Added `executeAsync` overload to accept a `RemoveStoredJobRequest` object for HTTP transmission instead of a simple empty `StoredJob_Remove` object.
3. **Device Service Layer (`IDeviceCopyJobService.java` / `StandardDeviceCopyJobService.java`)**
   - Implemented `deleteStoredJob` method overload to accept the `RemoveStoredJobRequest` request parameter.
4. **Business Logic Layer (`CreatingCopyJobState.java`)**
   - Inside `processDeleteJobIntent`, parses `reqParams.getJobCredentialsAttributes()` to extract the password when the job has a password set (not NONE type).
   - Sets the extracted password via `setJobPassword()` on the `RemoveStoredJobRequest` and passes it to `copyJobService.deleteStoredJob()`.

## Verification
- After modification, build and unit tests (`Test-WorkpathServices` sub-packages) completed successfully.
- The flow is now established where deletion (Remove) is performed with the correct password without 403 authorization errors.
