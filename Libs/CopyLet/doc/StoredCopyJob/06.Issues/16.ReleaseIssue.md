# 16. Release Issue — Client Recognizes Release as Store

## Symptoms

- Even after pressing the Release button, the client app displays **"Store Copy completed and stored"** toast message
- "Scanning" text is displayed during progress (E2 reports reading from internal storage as scanning)
- Number of printed pages, copies, and other output information is not displayed on the client
- At the E2 level, Release API succeeds (HTTP 201), and actual scan+print completes normally

## Root Cause (2 Stages)

### Cause 1: `createCopyJobBundle` uses the saved CopyAttributes' STORE mode as-is

```
processReleaseJobIntent
  → Load savedCopyAttributes (value saved during original STORE job)
  → releaseReader = new CopyAttributesReader(savedCopyAttributes)
  → releaseReader.getJobExecutionMode() == STORE
  → createCopyJobBundle() → mCopyJobData.setJobExecutionMode(STORE)
```

**1st fix**: Clear with `setJobExecutionMode(null)` after `createCopyJobBundle` call in `processReleaseJobIntent`

### Cause 2: `CopyJobCompletedState` reverts null mode back to STORE

```
CopyJobCompletedState.processCompletedJob()
  → copyJobData.getJobExecutionMode() == null  (value cleared by 1st fix)
  → copyJobData.setJobExecutionMode(STORE)     ← set back to STORE!
  → Client sees STORE at completion time and displays "Store Copy completed and stored"
```

### Correct Design

| Job Type | JobExecutionMode | Meaning |
|-----------|-----------------|---------|
| STORE (save) | `STORE` | Scan only, save to storage |
| RELEASE (output) | `null` | Print stored data to printer |
| NORMAL (regular copy) | `NORMAL` | Regular copy job |

## Changes

### 1. CreatingCopyJobState.java — `processReleaseJobIntent`

```java
// Before (1st fix state)
Bundle releaseJobBundle = createCopyJobBundle(clientVersion, releaseReader);
JobInfo jobInfo = releaseJobBundle.getParcelable(ILetObserver.Keys.KEY_JOB_INFO);
if (jobInfo != null) {
    if (releaseReader == null) {
        jobInfo.setJobName("[Stored Job]");
    }
    if (jobInfo.getJobData() instanceof CopyJobData) {
        ((CopyJobData) jobInfo.getJobData()).setJobExecutionMode(null);
    }
}

// After (2nd fix)
Bundle releaseJobBundle = createCopyJobBundle(clientVersion, releaseReader);
JobInfo jobInfo = releaseJobBundle.getParcelable(ILetObserver.Keys.KEY_JOB_INFO);
if (jobInfo != null) {
    jobInfo.setJobName(releaseReader != null
            ? releaseReader.getStoreJobName() : "[Stored Job]");
    if (jobInfo.getJobData() instanceof CopyJobData) {
        ((CopyJobData) jobInfo.getJobData()).setJobExecutionMode(null);
    }
}
```

Change: `jobName` setting logic changed from conditional on `releaseReader == null` to always set. Uses "[Stored Job]" if null, otherwise uses original stored job name.

### 2. CopyJobCompletedState.java — `processCompletedJob`

```java
// Before
@Override
protected void processCompletedJob(BaseJobIntentServiceStateMachine stateMachine) {
    try {
        JobInfo jobInfo = stateMachine.getJobInfo();
        if (jobInfo != null && jobInfo.getJobData() instanceof CopyJobData) {
            CopyJobData copyJobData = jobInfo.getJobData();
            if (copyJobData.getJobExecutionMode() == null) {
                copyJobData.setJobExecutionMode(CopyAttributes.JobExecutionMode.STORE);
            }
        }
    } catch (Exception ignored) {
    }
    // Delete stored job after copy completes (Release only)
    ...
}

// After
@Override
protected void processCompletedJob(BaseJobIntentServiceStateMachine stateMachine) {
    // Note: JobExecutionMode is already set correctly by CreatingCopyJobState:
    //   STORE  → for stored copy jobs (scan to storage)
    //   null   → for release jobs (print from storage)
    //   NORMAL → for normal copy jobs

    // Delete stored job after copy completes (Release only)
    ...
}
```

Change: Removed null → STORE conversion logic. Since STORE/RELEASE/NORMAL modes are set correctly in CreatingCopyJobState, re-setting in CompletedState is unnecessary.

- **Problem**: `JobData.jobName=StoreJob_20260414204014` ← Store name persisted
- **Problem**: Client `"Store Copy completed and stored"` ← due to STORE mode

### "Scanning" Text Issue
E2 CopyNotification reports `scanningActivity.started` during Release. This is because E2 reports the process of reading data from internal storage as scanning during a Release job. This is default E2 behavior and not an SDK code issue.
